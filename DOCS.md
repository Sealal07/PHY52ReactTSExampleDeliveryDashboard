# Часть 1. Как компоненты общаются? Пропсы (Props) и Функции обратного вызова (Callbacks)
Любое React-приложение состоит из компонентов. Компоненты можно представить как функции в математике: они принимают что-то на вход, обрабатывают это и выдают готовый HTML-код (UI) на выходе.

То, что приходит компоненту на вход, называется Props (свойства). В React управление данными подчиняется принципу однонаправленного потока данных (One-Way Data Binding) — данные всегда строго идут сверху вниз: от родительского компонента к дочернему.



## 1.1 Передача данных вниз (Props)
Родительский компонент передает конфигурацию и состояние дочернему через атрибуты. Главное правило: пропсы являются неизменяемыми (immutable) для того компонента, который их получил. Компонент-потомок имеет право только прочитать их и отобразить, но не может изменить напрямую.



## 1.2 Передача событий вверх (Callbacks)
Что делать, если в дочернем компоненте происходит событие (например, клик по кнопке), о котором должен узнать родитель? Сам потомок не имеет полномочий менять глобальное состояние или логику приложения.

Для этого используется механизм функций обратного вызова (Callbacks):

Родитель передает дочернему компоненту функцию в качестве пропса (обычно такие пропсы называют с приставки on..., например onSelectedOrder).

Дочерний компонент при наступлении события (клик, ввод текста) вызывает эту функцию и, если нужно, «выплевывает» в нее аргументы (например, ID элемента).

Родитель перехватывает этот вызов, получает аргументы и выполняет основную логику (например, открывает модальное окно или меняет состояние).

🔗 Официальная документация: [React: Передача пропсов в компонент](https://react.dev/learn/passing-props-to-a-component) | [React: Реагирование на события](https://react.dev/learn/responding-to-events)



# Часть 2. Почему нельзя просто изменить массив? Иммутабельность состояния
В React управление состоянием (State) строится на строгой концепции Иммутабельности (Неизменяемости). Вместо прямой модификации (мутации) существующих объектов или массивов в памяти, разработчик обязан всегда создавать их абсолютно новые копии с измененными свойствами.



## 2.1 Почему React «ленивый»?
Чтобы понять, что на экране нужно что-то перерисовать, React выполняет поверхностное сравнение (Shallow Comparison). Он проверяет старое состояние и новое по ссылке в памяти: oldState === newState.

Если вы мутируете элемент внутри массива напрямую (например, state.data[0].status = 'Delivered'), сама ссылка на массив (коробка, в которой лежат данные) в оперативной памяти остается прежней. React посмотрит на ссылку, скажет: «Коробка та же самая, ничего не изменилось» — и не обновит экран.



## 2.2 Как правильно обновлять данные?
Чтобы React зафиксировал изменения, мы должны создать новую ссылку (новую коробку). Для этого используются методы JavaScript, которые не мутируют старый массив, а возвращают новый:

Метод .map(): Идеален для обновления. Он берет старый массив, перебирает каждый элемент, применяет к нему изменения и складывает результат в абсолютно новый массив.

Spread-оператор (...): Используется для копирования объектов и массивов. Запись вида {\...order, status: 'In_Transit'} расшифровывается так: «Создай новый объект, скопируй в него все старые свойства объекта order, но свойство status перезапиши».

🔗 Официальная документация: [React: Обновление объектов в состоянии](https://react.dev/learn/updating-objects-in-state) | [React: Обновление массивов в состоянии](https://react.dev/learn/updating-arrays-in-state)



# Часть 3. Что такое React Context и зачем он нужен?
Представьте структуру компонентов как большую семью: Прадедушка, Дедушка, Папа и Сын. Прадедушка хочет подарить Сыну часы.



## 3.1 Проблема: Props Drilling
В обычном React Прадедушка обязан передать часы Дедушке, Дедушка — Папе, а Папа — Сыну. Это называется Props Drilling (прокачивание пропсов). Дедушке и Папе эти данные вообще не нужны, они не используют их, а просто работают «курьерами». Это засоряет код и усложняет поддержку.



## 3.2 Решение: Context API
React Context — это встроенный механизм, создающий независимое «облачное хранилище» (или сейф) прямо над деревом компонентов.

Паттерн состоит из трех частей:

Контекст (createContext): Сам контейнер для данных. Сюда можно передать значение по умолчанию (например, undefined), которое сработает, если компонент попытается взять данные, находясь вне этого хранилища.

Провайдер (Context.Provider): Компонент-обертка. Он «транслирует» данные (value) для всех компонентов ниже по дереву. При обновлении value все компоненты, подписанные на контекст, автоматически перерисовываются.

Потребитель (useContext): Хук, с помощью которого любой компонент на любой глубине вложенности может подойти к «сейфу» и взять данные напрямую, минуя промежуточных родителей.

🔗 Официальная документация: [React: Глубокая передача данных с помощью Context](https://react.dev/learn/passing-data-deeply-with-context) | [React Reference: useContext](https://react.dev/reference/react/useContext)



# Часть 4. Разбор синтаксиса [TypeScript: Размеченные объединения](https://www.typescriptlang.org/docs/handbook/unions-and-intersections.html) и индексация
TypeScript делает разработку безопасной, помогая отлавливать ошибки еще на этапе написания кода, а не в браузере у пользователя. Рассмотрим два важных инструмента.



## 4.1 Размеченные объединения (Discriminated Unions)
Это паттерн, позволяющий строго разграничить состояния системы. Для этого используется оператор | (вертикальная черта — логическое ИЛИ для типов).

Представьте тип состояния загрузки данных:

TypeScript
type DeliveryState = 
    | { status: 'LOADING' }
    | { status: 'SUCCESS'; data: DeliveryOrder[] }
    | { status: 'ERROR'; message: string }
Здесь свойство status является дискриминантом (маркером). Если TypeScript видит, что status === 'SUCCESS', он автоматически выполняет сужение типов (Type Narrowing) и разрешает нам использовать массив data. Если статус равен 'ERROR', поле data будет недоступно, зато появится message. Это полностью исключает случайные ошибки вроде «Cannot read property 'map' of undefined».



## 4.2 Доступ к типу по индексу (Indexed Access Types)
Иногда нам нужно сослаться на тип конкретного свойства внутри уже существующего интерфейса. Синтаксис выглядит так: DeliveryOrder['status'].

Вместо того чтобы вручную дублировать тип статуса (например, писать, что это строка), мы говорим компилятору: «Посмотри в структуру DeliveryOrder, найди там поле status и скопируй его тип сюда». Если в будущем набор статусов изменится в одном главном интерфейсе, во всем приложении типы обновятся автоматически.

🔗 Официальная документация: [TypeScript: Размеченные объединения](https://www.typescriptlang.org/docs/handbook/unions-and-intersections.html) | [TypeScript: Типы с доступом по индексу](https://www.typescriptlang.org/docs/handbook/2/indexed-access-types.html)



# Часть 5. Шаблонные (Generic) компоненты
Generics (Обобщения) — это инструмент TypeScript, позволяющий создавать компоненты, которые могут работать с самыми разными типами данных, не теряя при этом строгой проверки типов (в отличие от опасного типа any).



## 5.1 Переменная типа <T>
Аналогия: Представьте чертеж универсальной коробки для вещей. На чертеже написано: «Коробка для предметов типа <T>».

Если при сборке коробки мы укажем вместо T тип Книга, коробка станет строго книжной.

Буква T — это просто переменная для типа (как x в уравнениях). По соглашению используют T (от слова Type), но там может быть написано любое слово.

В React это позволяет делать универсальные UI-компоненты: списки, таблицы, селекты. Компонент списка принимает массив элементов типа T[] и функцию их отрисовки renderItem: (item: T) => ReactNode. Ему все равно, что внутри — список пользователей или список заказов, TypeScript сам поймет конкретный тип в момент передачи пропсов.

```ts
// Базовый синтаксис generic-функции компонента:
export const GenericList = <T,>({ items, renderItem }: GenericListProps<T>) => { ... }
```


## 5.2 Зачем нужна запятая в <T,>?
Это синтаксический хак для работы внутри .tsx файлов (где смешан код TypeScript и HTML-подобные теги JSX).
Если написать просто <T>, парсер решит, что вы открываете обычный HTML-тег (как <div> или <span>), не найдет закрывающего тега и выдаст ошибку. Запятая <T,> явно говорит компилятору: «Это не HTML, это объявление дженерик-переменной типа».

🔗 Официальная документация: [TypeScript: Руководство по Generics](https://www.typescriptlang.org/docs/handbook/2/generics.html)



# Логическая цепочка архитектуры
Все эти пять тем связываются в единую архитектурную цепочку:

Мы проектируем надежные типы состояний (Discriminated Unions).

Записываем их в защищенное от случайных мутаций хранилище (Immutability).

Доставляем эти данные в любую точку приложения без лишней «курьерской» работы (Context API).

Настраиваем гибкие, переиспользуемые UI-компоненты для отображения этих данных (Generics).

Возвращаем сигналы о действиях пользователя обратно к бизнес-логике через управляемые события (Callbacks).
